# Self-Evaluation

## 1. Хэрэв шалгалт өнөөдөр болбол би энэ кодыг өөрөө бичиж чадах уу?

**Хэсэгчлэн** — тайлбар:

Spring Boot-ийн layered architecture, REST endpoint,
JPA entity зэргийн ерөнхий бүтцийг ойлгосон.
Controller, Service, Repository-ийн үүрэг, хоорондын
харилцааг тайлбарлаж чадна.

Гэхдээ бүх кодыг санахгүй — ялангуяа JPA annotation-
уудын дэлгэрэнгүй (`@JoinColumn`, `@ManyToOne` гэх мэт),
Maven pom.xml-ийн dependency тохиргоо, Spring Boot-ийн
autoconfiguration механизмыг бүрэн гараар бичих хэцүү.

Unit test-үүдийн Mockito syntax (`when`, `thenReturn`,
`verify`) -г ойлгосон ч гараар бичихэд баримт бичиг
хэрэгтэй болно.

## 2. Дахин хийнэ гэвэл юуг өөрөөр хийх вэ?

**Lombok-г эхнээс орхих:** Lombok-тэй холбоотой
асуудалд их цаг зарцуулсан. Дахин хийвэл эхнээс
гараар getter/setter бичих байсан.

**PostgreSQL эхнээс суулгах:** H2 нь development-д
тохиромжтой ч production-д PostgreSQL ашиглах тул
эхнээс суулгаж тохируулах байсан.

**Commit-үүдийг жижиг, тодорхой хийх:** Зарим
commit-д хэт их файл нэгэн зэрэг орсон (жишээ:
"whole project" commit). Дахин хийвэл feature
бүрийг тусдаа commit болгох байсан.

**AI session log-ийг тухай бүрд хадгалах:**
Session дуусаагүй байхад log бичих байсан —
сүүлд санаж бичихэд зарим дэлгэрэнгүй мартагдсан.

## 3. Энэ туршлагаас юу сурсан бэ?

**AI-д итгэхгүй байх чухал:** Lombok асуудлаас
харахад AI өөртөө итгэлтэй байсан ч буруу байсан.
"Verify, don't trust" зарчим практикт маш чухал
гэдгийг ойлгосон.

**Алдааны мессежийг өөрөө уншиж сурах:** Maven-ий
compile алдааг AI-д шууд өгөхийн оронд өөрөө уншиж
ойлгохыг хичээсэн нь Java-ийн compile механизмыг
илүү сайн ойлгоход тусалсан.

**AI workflow-г зохион байгуулах чадвар:** Spec →
Generate → Review → Integrate дарааллыг дагаж
ажиллах нь AI-тай хамтрах хамгийн үр дүнтэй арга
гэдгийг ойлгосон.

**Кодлогчийн үүрэг өөрчлөгдөж байна:** AI код
бичихэд туслах боловч архитектурын шийдвэр,
алдааны шалтгааныг ойлгох, аюулгүй байдлыг
шалгах үүрэг хүний гарт үлдэж байна.v
